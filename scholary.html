Project abstract

Requirements

Create a Web application/platform able to model and manage – including operations like query, visualize, recommend,... – the knowledge about the provenance of each online newspaper article, which content could be textual or multimedia (pictures, illustrations, presentations, podcasts – audio files available for streaming, videos,...), possibly available in different languages and structured by using meta-data/classification models like DCMI, ITPC Standards and Social Semantic Web Thesaurus. Additional (meta)data could be provided by DBpedia and/or Wikidata. A SPARQL endpoint – exposing RDFa and JSON-LD including Creative Work concept from schema.org – is also necessary to provide access to interesting knowledge (e.g., list fresh editorials concerning a specific topic, provide the articles under 4000 words written in English or Spanish about the international IT contests, offer the title + description of the investigations and/or documentaries posted on Web by the Romanian journalists). For inspiration, study .Europeana Stories and The London Gazette. Explore also News Services APIs

Propoused solution

We propose a multi-service application which will aggregate a multitude of micro-service and will provide a user-friendly interface capable.

The interface should be used to input a link of an OpenAPI and a query formultated in natural language.

We will have a central API point that will play the role of the static delivery hub and of a gateway for the rest of the services. The service we will use are:

  • Transalation Service

    Will take the natural language query and will translate it to english. English is language used in most OpenAPi descriptions.

  • Preprocessing Service

    Will interrogate the OpenAPI and process the response in order to extract the keywords. This service will attempt to create a mapping from english words to API routes.

  • NaturalLanguage Service

    Will use the information provided by Translation Service and Preprocessing Service in order to compute a query. We will try to use the knowledge we have in order to create a API call and return it to the user.

API routes

Gateway API

    • Querry parameters :
      • OpenApiSchemaUrl
      • NaturalLanguageQuery
      • NaturalLanguageQueryLanguage
    • Body: empty
    • Reasponse:
